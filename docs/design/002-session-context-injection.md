# 002 — Session Context Injection

## Status

Current. Implemented in `scripts/session-start.sh`.

## Summary

The session-start hook injects a minimal context payload into every agent session so agents have project awareness without reading the full codebase. The design prioritizes low token cost — full documents are read on demand, not injected upfront.

## Decision

Inject four lightweight layers at session start. Skill routing is always injected (it's the agent's guide to when each `/ship:*` skill applies). Learnings are injected in full. All other documents use pointers or compact indexes.

## Design

### Four Injection Layers

| Layer | Source | What's Injected | Token Cost |
|-------|--------|-----------------|------------|
| 1. Skill routing | Inline in script | Trigger-condition table for all `/ship:*` skills | Low (~300 tokens, fixed) |
| 2. Learnings | `.learnings/LEARNINGS.md` | Full content | Medium (grows with project) |
| 3. Docs index | `docs/DOCS_INDEX.md` | Compact table (#, Status, Name, Description, Last Modified, Path) | Low (one row per doc) |
| 4. Visual design | `DESIGN.md` | One-line pointer | Minimal |

### Layer 1: Skill routing (always injected)

A compact table mapping trigger conditions to `/ship:*` skills. This solves the under-triggering problem — skill descriptions in the `available_skills` list compete with 100+ other skills for attention. Injecting a routing table into the session context ensures the agent knows when to reach for each skill. The table uses quoted trigger phrases ("plan this", "ship it") so the agent can match user intent to the right skill. A disambiguation rule at the bottom handles the `/ship:auto` vs individual-skill ambiguity.

### Layer 2: Learnings (full injection)

Learnings are injected in full because they contain rules that agents must follow on every action. Skipping a learning can cause bugs or architectural violations.

### Layer 3: Docs Index (compact table)

`docs/DOCS_INDEX.md` is auto-generated by `scripts/generate-docs-index.sh` from design doc frontmatter. The index includes:

- **#** — doc number (zero-padded 3 digits)
- **Status** — trust signal (current, draft, partially-outdated, not-implemented)
- **Name** — doc title
- **Description** — one-line summary for relevance filtering
- **Last Modified** — ISO date of last update
- **Path** — link to the doc by number

Superseded docs are excluded from the index — their guidance is replaced by another doc.

### Layer 4: Visual Design (pointer only)

When `DESIGN.md` exists at project root, the hook injects one sentence:

> DESIGN.md (visual design system) exists at project root. When writing frontend code, read it first.

The full design system (colors, typography, spacing, components) is read on demand by agents that need it.

### Output Format

The hook outputs JSON for the harness:
- Claude Code: `hookSpecificOutput.additionalContext`
- Cursor: `additional_context`

All parts are concatenated with `---` separators and JSON-escaped via `jq` (with `python3` fallback).

### Session ID Tracking

The hook also captures the session ID to `.ship/session-id.local` for session-isolated features like the auto verifier stop loop.

## Boundaries

- Skill routing and learnings are the only layers injected in full — all others use pointers or indexes.
- Skill routing is always injected, even if no context files exist.
- The skill routing table must stay under ~300 tokens — one row per skill, trigger phrases only.
- The docs index must never include superseded docs.
- The visual design pointer must never include actual design tokens — agents read the file when needed.
- The hook must not fail if `jq` is unavailable — it falls back to `python3`.
